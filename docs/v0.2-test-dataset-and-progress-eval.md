# v0.2 候选方案:40 天真实数据集 + 总体进度评估

> 写于 2026-04-21 凌晨。用户使用真实假数据集验证 v0.1 功能完整性,同时设计整体状态评估新功能。

---

## 一、40 天数据集设计

### 覆盖
- **时间**:2026-03-12 → 2026-04-20(40 天)
- **对应**:当前 408 plan 的"基础重建"阶段**前 40 天**(假设 plan 是 3 月起开始的二战考生)
- **用户画像**:考研二战,基础扎实但自律起伏明显

### 三段式完成率(仿真)

| 阶段 | 天数 | 完成率 | feeling 主流 | skipped 比例 |
|---|---|---|---|---|
| 新鲜期 (Day 1-14) | 14 | 80% | 4 | 10% |
| 疲劳期 (Day 15-28) | 14 | 55% | 3 | 30% |
| 回升期 (Day 29-40) | 12 | 70% | 3-4 | 15% |

### 真实度细节
- **周末偶尔高强度**:Day 6/13/20/27/34 共 5 天,每天 6-7 张任务
- **每周 1 天完全摆烂**:Day 10/18/25/32 共 4 天,全 skipped
- **feeling 极值**:每周随机 1 次 1-2(挫败)+ 1 次 5(爽)
- **连续 3 天低完成率窗口**:Day 15-17 设为≤40%(测试 T3 降量规则)

### 模块占比(按 408 规划)

| 模块 | 占比 | 典型 task 示例 |
|---|---|---|
| 高数 | 30% | 武忠祥基础班 ep5 + 课后 20 题 · 错题复习 |
| 数据结构 | 15% | 王道 4.2 二叉树遍历 · 编程实现 |
| 组成原理 | 15% | 王道 CO 第 3 章存储系统 + 框架图 |
| 翻译 | 10% | 2015 真题 Part 1 · 唐静拆句 |
| 词汇 | 10% | 墨墨 50 个新词 + 复习 |
| 阅读 | 10% | 剑雅 10 Test 1 Passage 1 |
| 其他 | 10% | 错题回顾 · 政治启动 |

**总任务数估算**:40 天 × 平均 5 张/天 = **200 条** + event 同步注入 ~400 条

### 生成脚本

位置:`backend/scripts/seed_fake_history.py`

规则:
- **不调 LLM**,纯模板 + 随机(0 token)
- 预定义每模块 8-10 个典型标题 + description 片段
- 所有假 task 标 `note="__seed_fake"`,**一键清**:
  ```sql
  DELETE FROM task WHERE note='__seed_fake';
  DELETE FROM event WHERE detail_json LIKE '%"fake":true%';
  ```

## 二、测试点(3 个)

### T1:针对性复盘规则(已在 v0.1-plus-dialog 实现)

**造数据**:Day 15 有 feeling=2 的高数 task,Day 18 有 skipped 的翻译 task

**期望**:今天 force_refresh 的任务中,**至少 1 张复盘任务**,title 含"复盘",rationale 明确指出哪个历史记录

**实测**(已在 v0.1-plus-dialog 验证):✅ 通过

### T2:动态 module 分布(已泛化)

**期望**:`/progress` 的 `module_distribution` 显示所有 21 个 408 子模块,占比按**本周实际完成时长**计算

**本次 40 天数据集会触发的真实分布**:高数 30% / 数据结构 15% / ... 等(匹配假数据模板比例)

### T3:连续低完成率降量(v0.1 prompt 有规则,未实测)

**造数据**:Day 15-17 连续 3 天完成率 < 40%(每天 5 张任务,只完成 1-2 张)

**期望**:Day 18 force_refresh 时,V3 应该感知到"连续 3 天完成率 < 50%",生成 3 张任务 + 总时长折半(~4.5h 而不是 9h)

**验证**:抓 V3 返回,check `len(tasks) == 3` 且 `sum(estimated_minutes) < 300`

## 三、新功能:总体进度评估

### 目的
当前 `/progress` 只看本周。用户需要 **"这 40 天我整体怎样"** 的视角。

### 后端:新端点 `GET /progress/full`

返回:
```json
{
  "plan_activated_at": "2026-03-12",
  "days_since_activation": 40,
  "overall_completion_rate": 0.68,
  "overall_avg_feeling": 3.4,

  "weekly_trajectory": [
    { "week_start": "2026-03-09", "rate": 0.82, "avg_feeling": 4.1, "tasks": 28 },
    { "week_start": "2026-03-16", "rate": 0.75, "avg_feeling": 3.8, "tasks": 30 },
    ...
    { "week_start": "2026-04-13", "rate": 0.68, "avg_feeling": 3.5, "tasks": 29 }
  ],

  "module_heatmap": {
    "高数": { "total_min": 1840, "done_rate": 0.72, "avg_feeling": 3.2 },
    "数据结构": { "total_min": 920, "done_rate": 0.85, "avg_feeling": 3.8 },
    ...
  },

  "status_assessment": "地基期后半段有疲劳迹象,建议下周二/四降量 30%。高数 feeling 整体偏低(3.2),考虑把新题时长减半,错题占比提到 60%",

  "milestone_predictions": [
    { "phase_name": "基础重建", "on_track": true, "confidence": 0.72, "completion_forecast": "2026-06-25(提前 5 天)" },
    { "phase_name": "强化突破", "on_track": null, "confidence": null }
  ]
}
```

### 关键算法(非 LLM)

- **overall_completion_rate** = done_tasks / total_tasks(plan 激活以来)
- **weekly_trajectory** = 按周分组的 rate 和 feeling 均值
- **module_heatmap** = group by task.module
- **milestone_predictions** = 线性拟合当前速度,预测能否在 phase.end_date 前完成 `phase.target_tasks`

### LLM 评级(可选,每 24h 缓存)

- `status_assessment` 用 V3,prompt 给它 `weekly_trajectory` + `module_heatmap` + `recent 7d tasks`
- 输出控制在 150 字内
- 缓存在 `learning_plan.latest_assessment`(新字段)+ `assessment_at` 时间戳
- 下次 24h 内直接读缓存

### 前端:`/progress` 页顶部新区块

```
[以下保留当前周视图]
距考试 132 天 ...
```

## 五、风险

| 风险 | 对策 |
|---|---|
| 假数据污染 V3 对"真实状态"的判断 | 所有假 task 标 `note="__seed_fake"`,一键删 |
| V3 评级不稳定 / 每次生成不同 | 加 24h 缓存,手动刷新覆盖 |
| 前端 200 task 性能 | 前端只拿聚合后 JSON,不拉原始 task |
| T3 降量规则实际 V3 是否会遵守 | 先手工测,不行就在 prompt 里加更硬的语言 |
